# Spec: scripts/chroma_seed

## Purpose

Seeds a ChromaDB collection from DuckDB movie records, generating two text variants per title:

- A human-readable catalog description.
- A semantic embedding description used as the document text for similarity search.

The script package writes retrieval data to ChromaDB (`titles` collection) and uses a local SQLite database as an operational store for resume state, per-title status, and failure diagnostics.

Batching is stage-based and OpenAI-client-oriented: for each logical batch of `N` titles, the script performs four operations in order: read `N` titles, generate `N` human descriptions via `N` concurrent single-title OpenAI chat-completions requests using `asyncio`, generate `N` embedding descriptions via `N` concurrent single-title OpenAI chat-completions requests using `asyncio`, and insert `N` records to ChromaDB as one batch write.

---

## Usage

```bash
uv run python scripts/chroma_seed/main.py [--batch-size N] [--limit N]
```

- `--batch-size N` - optional; records per batch (defaults to configured `BATCH_SIZE`).
- `--limit N` - optional; process only first `N` matching titles (development/testing).

Requirements:

- DuckDB must already be seeded (`back-end/data/imdb.duckdb` exists).
- ChromaDB must be reachable.
- An OpenAI-compatible chat-completions endpoint must be reachable (local hosted endpoints are supported via base URL).
- Text generation calls use either the OpenAI Python client or direct HTTP POST when `OPENAI_BASE_URL` points to `/chat/completions`.
- The script may prompt interactively when previous SQLite state is detected.

## Configuration

| Constant / Setting | Default | Description |
| --- | --- | --- |
| `DUCKDB_PATH` | `back-end/data/imdb.duckdb` | Source database for title selection |
| `SQLITE_PATH` | `data/chroma_seed.sqlite` | Local operational DB for state and failure logging |
| `CHROMA_HOST` | `localhost` | ChromaDB HTTP host |
| `CHROMA_PORT` | `8001` | ChromaDB HTTP port |
| `COLLECTION_NAME` | `titles` | Target ChromaDB collection |
| `BATCH_SIZE` | `20` | Default records per batch |
| `TEXT_GENERATION_PROVIDER` | `OpenAI-compatible API` | Provider transport used for text generation calls |
| `TEXT_GENERATION_MODEL` | `llama3.2:3b` | Model used for human/embedding-description generation |
| `OPENAI_BASE_URL` | `http://localhost:11434/v1/chat/completions` | OpenAI-compatible endpoint base URL |
| `OPENAI_API_KEY` | `any-api-key` | API key used by the OpenAI transport |
| `MAX_RETRIES` | `3` | Retry limit for text-generation calls |
| `HUMAN_MAX_TOKENS` | `200` | Max output tokens for human-description generation |
| `EMBEDDING_MAX_TOKENS` | `250` | Max output tokens for embedding-description generation |
| `INFERENCE_CONCURRENCY` | `BATCH_SIZE` | Max in-flight concurrent inference requests per phase |
| `MAX_CONSECUTIVE_TITLE_FAILURES` | `10` | Global safety stop threshold for consecutive failed titles |

Provider/model settings are loaded from environment variables or explicit module constants. The implementation routes text generation through chat-completions using one of two transports: the OpenAI Python client (base API URL) or direct HTTP POST (explicit `/chat/completions` URL). Requests use explicit output limits (`max_tokens`) and `temperature=0` to stabilize throughput. Inference batching is implemented by issuing concurrent single-title requests with `asyncio` and a concurrency cap of `min(INFERENCE_CONCURRENCY, current_batch_size)` (never below `1`). Embeddings are generated by ChromaDB from the provided document text.

## Module Layout

Implementation must be modular and placed under `scripts/chroma_seed/`.

Recommended structure:

- `scripts/chroma_seed/main.py` - CLI entrypoint and top-level orchestration.
- `scripts/chroma_seed/config.py` - constants, CLI defaults, and environment loading.
- `scripts/chroma_seed/duckdb_reader.py` - title counting and batched title selection queries.
- `scripts/chroma_seed/llm_client.py` - OpenAI client setup and asyncio concurrent prompt execution.
- `scripts/chroma_seed/chroma_writer.py` - Chroma collection setup and batched inserts.
- `scripts/chroma_seed/sqlite_store.py` - SQLite schema initialization and status/failure persistence.
- `scripts/chroma_seed/progress.py` - tqdm progress bars and runtime metrics formatting.
- `scripts/chroma_seed/models.py` - typed payload models used across modules.

Equivalent modular decomposition is acceptable, but all functionality must remain inside `scripts/chroma_seed/` rather than a single-file script.

## Title Selection Criteria

Source tables:

- `title_basics`
- `title_ratings` (joined on `tconst`)

Selection filters:

- `titleType = 'movie'`
- `averageRating > 7.5`
- `startYear > 2013`
- `startYear < 2024`

Selected fields:

- `tconst`
- `primaryTitle` (as `title`)
- `startYear`

Ordering for deterministic batching/resume:

```sql
ORDER BY tconst
```

## Execution Phases

1. **Initialize dependencies**. Connect to ChromaDB, open DuckDB in read-only mode, and open or create the SQLite operational database.

2. **Detect previous state**. If `SQLITE_PATH` contains prior run data and the process is interactive (`stdin` is a TTY), prompt the user to continue from the last successful title id or clear state and restart from the beginning. In non-interactive mode, continue automatically.

3. **Prepare Chroma collection**. Ensure collection `titles` exists. If the user chooses restart, delete and recreate the target collection before processing.

4. **Process one batch**. Fetch the next batch using deterministic ordering. On resume, select titles strictly after the last successful `title_id` (lexicographically by `tconst`) so already completed titles are skipped.

5. **Run human-description LLM batch**. Submit `N` independent human-description prompts (one per title) as concurrent OpenAI chat-completions requests using `asyncio` and collect `N` raw-text responses.

6. **Run embedding-description LLM batch**. Submit `N` independent embedding-description prompts (one per title) as concurrent OpenAI chat-completions requests using `asyncio` and collect `N` raw-text responses.

7. **Persist batch outputs**. Write `N` items to ChromaDB using batch insert APIs, then upsert per-title status to SQLite and record failures to SQLite with phase and error details.

8. **Repeat until complete**. Continue processing batches until all selected titles are handled.

9. **Print final statistics**. Report processed vs total titles, success vs failed counts, and average processing time per title.

Async scheduling note:

- The pipeline keeps deterministic logical stage ordering (human phase then embedding phase), while concurrent request completion order may vary within a phase.
- Persistence semantics remain tied to logical batch boundaries in this spec.

## ChromaDB Collection Schema

Collection name: `titles`

Each successful record must include:

| Chroma field | Type | Value |
| --- | --- | --- |
| `id` | string | `tconst` |
| `document` | string | embedding description text |
| `metadata.titleId` | string | `tconst` |
| `metadata.title` | string | `primaryTitle` |
| `metadata.startYear` | int | `startYear` |
| `metadata.human_description` | string | human description text |

`titleId` and `id` both map to `tconst`; duplication in metadata is intentional to support downstream filtering conventions.

The script provides `id`, `document`, and `metadata` to ChromaDB. Embedding vectors are computed by ChromaDB according to the collection's configured embedding function.

---

## SQLite Operational Schema

SQLite is not the primary retrieval store. It is used for run control, resume behavior, and diagnostics.

Required tables:

### `seed_records`

Tracks per-title processing state.

| Column | Type | Notes |
| --- | --- | --- |
| `title_id` | TEXT PRIMARY KEY | IMDB `tconst` |
| `title` | TEXT NOT NULL | Movie title |
| `start_year` | INTEGER | Year |
| `human_description` | TEXT | Populated on success |
| `embedding_description` | TEXT | Populated on success |
| `status` | TEXT NOT NULL | `pending` / `success` / `failed` |
| `last_error` | TEXT | Latest error if failed |
| `updated_at` | TEXT NOT NULL | ISO timestamp |

### `seed_failures`

Append-only failure log for post-run review.

| Column | Type | Notes |
| --- | --- | --- |
| `id` | INTEGER PRIMARY KEY AUTOINCREMENT | Row id |
| `title_id` | TEXT NOT NULL | IMDB `tconst` |
| `phase` | TEXT NOT NULL | `human_generation` / `embedding_generation` / `chroma_write` |
| `attempt` | INTEGER NOT NULL | Retry attempt number |
| `error_message` | TEXT NOT NULL | Captured exception message |
| `created_at` | TEXT NOT NULL | ISO timestamp |

Resume cursor source:

- The script derives resume position from `seed_records`.
- Last successful title id is `MAX(title_id)` where `status = 'success'`.
- If no successful rows exist, resume starts from the beginning of the ordered result set.

## Prompt Templates

Prompt construction requirements for OpenAI client execution:

- Keep the system prompt text byte-stable per prompt type (human vs embedding) across all batches in a run.
- Pass title/year per item in the user prompt message (`Film`, `Title`, `Year`) rather than in the system prompt template.
- Each prompt must represent exactly one title and remain independent from other titles in the same batch.
- Preserve deterministic item ordering from the selection query (`ORDER BY tconst`) when building the list of per-title prompts in a batch.
- Avoid injecting dynamic instructions, timestamps, or run-specific text into system prompts.
- Request and response bodies should be plain text for prompt content and generated output; do not require model-generated JSON structures.

Per-title response contract:

- Each concurrent request returns one raw-text description.
- The script maps responses to titles by request index or provider-level request identifiers, not by requiring `title_id` inside generated text.
- Empty or missing outputs are treated as failed title generations for retry and logging.

### Human Description Prompt

```text
You are generating a short film description for a catalog.

Write a clear, neutral summary of the film.

Requirements:
- 3-4 sentences
- 70-100 words
- No opinions, ratings, or promotional language
- No actor names
- Focus on premise, main characters, and central conflict
- Mention setting only if relevant
- Use natural, readable language

Output plain text only.

Film:
 - Title: {title}
 - Year: {startYear}
```

### Embedding Description Prompt

```text
You are generating a structured semantic description of a film for embedding and similarity search.

Rules:
- Be concise, factual, and information-dense.
- Do NOT include opinions, ratings, or subjective language.
- Do NOT use marketing phrases.
- Use consistent vocabulary.
- Output in plain text (no JSON, no markdown).
- Max 120 words.
- Use simple vocabulary. Avoid rare words and synonyms.
- Prefer canonical genre and theme labels.

Fill in all fields in this structure:
Genres:
Setting:
Themes:
Plot:
Characters:
Style:
Director:
Writer:
Starring:

Film:
 - Title: {title}
 - Year: {startYear}
```

## Batching and Retry Policy

- Process data in logical stage batches of size `BATCH_SIZE` (or `--batch-size`).
- For each logical title batch, submit up to `INFERENCE_CONCURRENCY` concurrent single-title OpenAI chat-completions requests with `asyncio` for the human phase, then repeat for the embedding phase.
- Default `INFERENCE_CONCURRENCY` equals `BATCH_SIZE`, so all titles in a batch are queried concurrently unless explicitly overridden.
- Include explicit `max_tokens` limits per request type to stabilize latency and throughput (`HUMAN_MAX_TOKENS`, `EMBEDDING_MAX_TOKENS`).
- Use ChromaDB batch insert APIs to write the full successful subset of the batch in one call.
- Retry failed title-level requests up to `MAX_RETRIES` (default `3`).
- If retries are exhausted, mark affected titles as failed, log failures, and continue with the next batch.
- If a batch response is partial, contains missing response slots, or returns empty/invalid text outputs, treat unmatched titles as failed for that attempt and include diagnostics in SQLite failure logs.
- Track consecutive failed titles across the full run and stop execution when failures reach `MAX_CONSECUTIVE_TITLE_FAILURES` (default `10`).
- LLM/chroma failure rows are currently written once per failed title after retries are exhausted, with `seed_failures.attempt` persisted as `MAX_RETRIES`.

## Progress Reporting

Use `tqdm` with:

- One overall progress indicator over total selected titles.
- One active batch indicator for the current batch.
- Runtime statistics include `processed/total`, `success`, `failed`, average processing time per title, elapsed time (time spent), ETA (estimated time to finish), description generation performance in seconds, ChromaDB save performance in seconds, and DuckDB query time in seconds.

Progress output should remain stable and readable during long runs.

---

## Error Handling

| Condition | Behavior |
| --- | --- |
| LLM title request fails | Retry up to 3 times; on final failure mark affected titles failed and log to SQLite |
| Chroma batch write/embedding computation fails | Retry up to 3 times; on final failure mark affected titles failed and log to SQLite |
| 10 titles fail in a row | Stop the run gracefully, print a clear stop reason, and exit non-zero |
| Chroma connection unavailable at startup | Exit before processing begins |
| DuckDB missing or unreadable | Exit with clear error |

Failure-attempt logging detail:

- `seed_failures.attempt` currently records the terminal failed attempt number (`MAX_RETRIES`) for each failed phase entry.

The script should fail fast on unrecoverable initialization errors and continue on per-title recoverable errors.
